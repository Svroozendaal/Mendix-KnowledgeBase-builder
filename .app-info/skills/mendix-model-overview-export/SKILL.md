# MENDIX MODEL OVERVIEW EXPORT

## Purpose

Use this skill to understand the v2.0 model overview export format that serves as input for the KnowledgeBase generation pipeline.

## Export Structure (v2.0)

A model overview export run produces the following folder structure:

```
<run-folder>/
  manifest.json                          # Run metadata, schema version, artifact list
  general/
    app-info.json          + .pseudo.txt  # App identity, security level, module count
    user-roles.json        + .pseudo.txt  # User roles and module role mappings
    all-modules.json       + .pseudo.txt  # All modules with metadata
    marketplace-modules.json + .pseudo.txt # Marketplace module details
  modules/
    <ModuleName>/
      domain-model.json    + .pseudo.txt  # Entities, attributes, associations, access rules
      flows.json           + .pseudo.txt  # Microflows, nanoflows, rules with execution order
      pages.json           + .pseudo.txt  # Pages, layouts, snippets with widgets and parameters
      resources.json       + .pseudo.txt  # Constants, scheduled events, Java/JS actions
```

Each file exists in two formats:
- **JSON** — machine-readable, complete data
- **Pseudocode (.pseudo.txt)** — AI-readable, formatted for interpretation

## How Exports Are Produced

Exports are generated by `MendixModelOverviewParser` (a C# parser) that reads a Mendix `.mpr` working dump (`mx dump-mpr` output). The parser:

1. Parses one working dump snapshot (no head comparison)
2. Builds object and resource indexes by `$ID`
3. Inventories domain model, flows, pages, and resources per module
4. Determines flow execution order using graph traversal
5. Detects flow-to-flow calls and builds a call graph
6. Outputs structured JSON + pseudocode per file category

## Key Conventions

- **Pseudocode flow ordering**: Flows are ordered by graph traversal from StartEvent nodes, with deterministic edge sorting for reproducible output.
- **Module classification**: Modules are classified as Custom, System, or Marketplace based on naming patterns and metadata.
- **Entity access rules**: Parsed with XPath constraints and member-level read/write permissions.
- **Page parameters**: Include entity type and required/optional status.

## Manifest Format

```json
{
  "schemaVersion": "2.0",
  "generatedAt": "ISO-8601 timestamp",
  "appName": "Application name",
  "artifacts": [
    { "path": "general/app-info.json", "type": "app-info" },
    { "path": "modules/SmartExpenses/domain-model.json", "type": "domain-model" }
  ]
}
```

## Downstream Consumer

The primary consumer of these exports is the **KNOWLEDGEBASE_CREATOR** agent, which delegates to interpretation skills to convert exports into an AI-navigable knowledge base.
